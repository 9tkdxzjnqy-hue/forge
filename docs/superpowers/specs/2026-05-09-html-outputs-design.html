HTML Outputs

Draft — pending review 2026-05-09 Brainstorming session

Decision

Dual-file approach: markdown is the canonical source, HTML is the generated presentation layer. For every discovery artefact (brief, assumption-log, personas, research-plan, breadboard, handoff), agents write .md first — clean, human-editable content, close to thought. They then generate .html from the markdown — the browsable presentation layer. The .md is the source of truth; the .html is derived from it.

Agent definition files in /product-team/ and /build-team/ also become HTML. DESIGN.md gains an .html companion. The pipeline dashboard stays HTML-only (it is a tracker, not a thought document).

Single-file markdown (no HTML pair): running-brief.md, decisions.md, and CLAUDE.md. These are agent coordination mechanisms or technical context files — their primary consumer is other agents, not humans. The pipeline dashboard provides the human-facing view of pipeline state.

CLAUDE.md — dual file approach: Claude Code reads files named CLAUDE.md specifically. The filename must stay .md for automatic pickup — this is the only technical constraint. Both the Forge repo and product repos keep CLAUDE.md as lean markdown, optimised for Claude Code reading at session start. Alongside it, Synthesis also produces CLAUDE.html — a rich, human-readable companion with full visual treatment. Same information, different density. CLAUDE.md is what Claude reads; CLAUDE.html is what you open in a browser.

Rationale

The dual-file approach is grounded in how thinking actually works. Text is one step from thought. When agents write markdown first, the content is transparent, editable, and graspable — you can stare at it, tweak it, grapple with it. If agents wrote HTML directly, humans would lose that connection. The raw idea becomes obscured behind markup. Markdown keeps ideas reachable. HTML makes them presentable. These are different jobs, and they need different files.

HTML is for sharing and presentation, not thinking. A brief in HTML is easier to read, navigate, and share across a team. A brief in markdown is easier to interrogate, edit, and reason about. The dual-file pattern gives you both without compromise — at the cost of a generation step that agents handle automatically.

The richness of the HTML format also shapes what agents produce. A breadboard rendered as HTML produces a flow diagram. The same instruction in markdown produces five tables. Format is not neutral — which is exactly why the .md source must remain the canonical form. The HTML is a consequence of the thinking, not a replacement for it.

The counterargument — that agents struggle to parse HTML — does not hold. Claude reads and writes HTML fluently. The token cost is higher but acceptable given the 1M context window and the measurable improvement in human engagement with outputs.

Hands off to → Breadboard (The Tracer)

The agent card (name, mode, reads, produces, handoff) is always visible. Full instructions are collapsible — default collapsed for human browsing, read in full by Claude when acting as the agent. The "What you never do" section is always visible, never buried in the collapsible — it defines framework violations and must be immediately readable.

An SVG interaction diagram accompanies each agent card showing pipeline position and lateral connections (Observer, Researcher, etc.).

Pipeline Dashboard

One file per idea: output/[idea-name]/pipeline-dashboard.html. Two modes in one file.

Live mode

Updated after each agent run. Shows current pipeline position, each completed agent with date and linked outputs, active agent with status dot, pending agents greyed out, open blockers surfaced prominently.

Milestone snapshots

At each phase transition, the Delivery Manager rewrites the dashboard with a new timestamped block. Blocks accumulate — the dashboard becomes a project timeline. Updates are milestone-gated (2–4 per project) and always written by the same agent.

Updates happen at milestone boundaries only (2–4 per project, not after every agent run), and ownership is always the Delivery Manager — making the HTML rewrite pattern straightforward.

Intellectual Standards — Framework-wide Behavioral Baseline

A new file: skills/intellectual-standards.md. Referenced by every agent. Default posture for all agent interactions.

1

Lead with the counterargument

Before supporting any position, steelman the strongest opposing view. Do not validate premises before examining them.

2

Explicit confidence levels

Every factual or analytical claim carries a marker: (high) / (moderate) / (low) / (unknown). Do not imply certainty you do not have.

3

No anchor on human framing

Generate your own independent assessment before comparing to the brief or running context. Do not let prior framing constrain your analysis.

4

Non-capitulation rule

If the human pushes back without new evidence or a superior argument, restate your position. Update only when genuinely persuaded. Social pressure is not a reason to change a view.

5

Negative conclusions are valid output

"This assumption is wrong," "this idea has a fatal flaw," "do not build this" are acceptable conclusions. Do not soften them into suggestions.

6

No disclaimers, no moralising

State conclusions directly. Do not preface with "it's important to consider" or equivalent hedges unless explicitly asked.

7

Accuracy is the success metric, not approval

An agent that makes the human uncomfortable by being right is doing its job. An agent that makes the human comfortable by being wrong is failing.

8

Surgical Changes (Engineer only)

Every changed line must trace to the assigned task. Do not improve adjacent code — mention it, don't touch it. Remove only the orphans your changes created. Match existing style even if you'd do it differently. The scope of a task is a constraint, not a suggestion.

The instruction to "make answers as long and detailed as possible" from the source post is explicitly excluded. Length is calibrated to what the human needs to make a decision — not maximised.

4
