# Skills · Frameworks

Reference material for discovery agents. Use these frameworks
to structure thinking — do not follow them mechanically.

---

## Jobs-to-be-done (JTBD)

The core idea: people don't buy products, they hire them to do
a job. Understanding the job is more useful than understanding
the product category.

**Jobs Story format:**
"When [situation], I want to [motivation], so I can [outcome]."

Good: "When I'm preparing for a meeting I didn't initiate,
I want to quickly understand the context and stakes,
so I can contribute meaningfully without appearing unprepared."

Bad: "When I need information, I want to find it fast,
so I can be productive." (Too vague — could describe any product)

**Three job dimensions:**
- Functional: the practical task to be accomplished
- Emotional: how the user wants to feel (or avoid feeling)
- Social: how the user wants to be perceived by others

**Sub-jobs:** The primary JTBD often contains 2–5 sub-jobs.
Identifying sub-jobs reveals where a product can win on dimensions
a competitor hasn't addressed.

## Opportunity Solution Tree (OST)

Framework for connecting desired outcomes to potential solutions
without jumping to solutions prematurely.

**Structure:**
```
Outcome (the metric you want to move)
  └── Opportunity 1 (an unmet need or pain point)
        └── Solution A (a way to address the opportunity)
        └── Solution B
  └── Opportunity 2
        └── Solution C
        └── Solution D
```

**Rules:**
- The outcome is a metric, not a feature
- Opportunities are user problems, not product features
- Solutions are specific and testable
- One opportunity can have multiple solutions — the OST helps
  you see which opportunities have the most solution options
  and which are most worth addressing

## HEART Framework (metrics)

For defining what success looks like:

| Dimension | Definition | Example metric |
|-----------|------------|----------------|
| Happiness | User satisfaction | NPS, satisfaction survey |
| Engagement | Frequency and depth of use | DAU, session length |
| Adoption | New users or features | Signup rate, feature uptake |
| Retention | Users who return | 30-day retention, churn |
| Task success | Can users complete the job | Completion rate, time-on-task |

**North star selection:** Choose the one HEART dimension most
aligned with the core JTBD. The north star metric should be the
number that, if it goes up, you are confident the product is
delivering real value.

## Problem statement format

A sharp problem statement has four components:
1. Who has the problem (specific user)
2. What they are trying to do (the job)
3. What currently gets in the way (the obstacle)
4. Why it matters (the cost of the problem remaining unsolved)

**Format:**
"[User] trying to [job] currently [obstacle], which means [cost]."

## Assumption types

When logging assumptions, classify by type:

- **Desirability:** Do users want this? Will they change behaviour?
- **Viability:** Does this work as a business or product?
- **Feasibility:** Can it be built? Can it be built well enough?
- **Usability:** Can users figure out how to use it?
- **Ethical:** Does this create risks or harms worth considering?
