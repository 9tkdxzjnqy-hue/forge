Two approaches to HTML adoption in Forge

The scope is settled — everything goes HTML. These differ in design philosophy.

A

Rich documents — each file is self-contained

Every HTML file is a standalone document with its own visual logic. The brief has tabs and a summary header. The breadboard has an SVG wiring diagram. Agent files have a visual card + expandable instructions. Each file is designed for its content.

For

  • Maximum richness per artefact
  • Each file designed for what it actually shows
  • Agents have freedom to choose the right visual representation

Against

  • Less visual consistency across files
  • Harder to template — agents need more judgment
  • Some files may end up richer than others
B

Design system — shared visual language first

Define a Forge design system: colour palette, typography, section components, diagram conventions. Every file — outputs, agent definitions, dashboards — uses this system. Brief, breadboard, and agent cards all look like they belong to the same product.

For

  • Consistent, professional feel across everything
  • Agents follow a shared component library
  • Dashboard, brief, and agent files feel unified

Against

  • More upfront work — design system before any files
  • Risk of constraining agents' ability to represent content well
  • System needs maintenance as new artefact types emerge
My recommendation
B — but lightweight. A minimal shared CSS file (colours, typography, section structure) embedded by agents in every output. Not a full component library — just enough to make everything feel coherent. Agents still have freedom within the system for content-specific visuals (SVG diagrams, tables, cards). The design system is a floor, not a ceiling.
